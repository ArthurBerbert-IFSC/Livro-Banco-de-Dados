1.1                     Dados e Informaes: o ponto de partida

Antes de estudarmos os bancos de dados espaciais, fundamental compreender a diferena entre dado e informao, pois ela orienta a forma como interpretamos a realidade geogrfica.

Um dado um valor isolado e bruto, como "populao = 5231" ou "temperatura = 32C". Esses nmeros, sozinhos, no respondem a perguntas analticas. J a informao surge quando os dados so organizados e contextualizados, permitindo concluir, por exemplo, que "o bairro X apresenta maior densidade populacional que o bairro Y".

O papel dos bancos de dados justamente transformar dados dispersos em informao estruturada. No geoprocessamento, trabalhamos com grandes volumes de registros coordenadas, atributos, imagens e medies ambientais. Sem organizao, esses dados seriam apenas fragmentos soltos. O banco de dados fornece a base para consultas, cruzamentos e anlises que do sentido ao material coletado.

Essa transformao segue uma hierarquia conceitual bastante utilizada: dado → informao → conhecimento → deciso. Em cada etapa, os registros brutos ganham significado, permitindo testar hipteses e chegar a decises fundamentadas.

Esse processo essencial para superar o achismo.

Afirmaes como "a regio Norte mais populosa que a regio Sul" ou "o Rio Itacorubi o mais poludo" permanecem como opinies at serem confrontadas com nmeros. O banco de dados possibilita verificar tais hipteses com evidncias, transformando especulao em conhecimento fundamentado.

1.5                    Sistemas de Gerenciamento de Bancos de Dados (SGBD).

Um Sistema de Gerenciamento de Banco de Dados (SGBD) um conjunto de softwares que permite aos usurios interagirem com um banco de dados. Ele atua como uma interface entre o usurio e o banco de dados, facilitando a criao, o acesso, a manipulao e o gerenciamento dos dados.0F[1]

1.5.1                 Funes de um SGBD:

Um SGBD desempenha diversas funes essenciais para o funcionamento eficiente e seguro de um banco de dados:

Armazenamento de dados: O SGBD responsvel por armazenar os dados de forma organizada e eficiente, utilizando estruturas de armazenamento adequadas ao tipo de dado. No contexto do geoprocessamento, isso envolve armazenar tanto os dados alfanumricos (atributos) quanto os dados geomtricos (localizao, forma).

Recuperao de dados: O SGBD permite recuperar os dados armazenados de forma rpida e eficiente, atravs de consultas e filtros. No geoprocessamento, isso permite buscar informaes espaciais com base em critrios como localizao, atributos geogrficos ou relaes topolgicas.

Atualizao de dados: O SGBD permite modificar os dados armazenados, inserindo novos registros, alterando registros existentes ou excluindo registros. No geoprocessamento, isso fundamental para manter os dados geogrficos atualizados, refletindo as mudanas no mundo real.

Segurana de dados: O SGBD oferece mecanismos para proteger os dados contra acessos no autorizados, perdas ou danos. Isso inclui controle de acesso, autenticao de usurios, criptografia e backups. No geoprocessamento, a segurana dos dados geogrficos crucial, especialmente quando se trata de informaes confidenciais ou estratgicas.

Controle de concorrncia: O SGBD gerencia o acesso simultneo de mltiplos usurios aos dados, garantindo que as operaes no interfiram umas nas outras e que os dados permaneam consistentes. No geoprocessamento, isso importante quando vrios profissionais trabalham no mesmo projeto, acessando e modificando os mesmos dados geogrficos.

Integridade de dados: O SGBD garante que os dados armazenados sejam precisos, consistentes e confiveis, aplicando regras e restries para evitar erros e inconsistncias. No geoprocessamento, isso fundamental para garantir a qualidade dos dados geogrficos e a preciso das anlises espaciais.

1.7                    Tipos de SGBD e suas caractersticas:

Existem diferentes tipos de Sistemas de Gerenciamento de Bancos de Dados (SGBD), cada um com caractersticas e funcionalidades distintas, projetados para atender a diferentes necessidades de armazenamento e manipulao de dados. Podemos agrup-los em duas categorias principais: SGBDs que seguem o modelo relacional e aqueles que adotam modelos no relacionais.

1.7.1                 SGBDs Relacionais: Estrutura e Organizao dos Dados

Os SGBDs relacionais, como PostgreSQL, MySQL e SQLite, so construdos sobre o modelo relacional, que se destaca pela organizao dos dados em tabelas bem estruturadas. Imagine uma planilha eletrnica: cada tabela composta por linhas, que representam os registros ou entidades do mundo real, como pessoas, cidades ou equipamentos, e colunas, que definem os campos ou atributos que descrevem essas entidades, como nome, localizao ou data de criao. Essa forma de organizao facilita a consulta, o cruzamento e a integridade dos dados, tornando o modelo relacional ideal para diversas aplicaes, inclusive no geoprocessamento.

A grande fora dos SGBDs relacionais reside na capacidade de estabelecer relaes bem definidas entre essas tabelas. Isso feito atravs do uso de chaves primrias, que identificam de forma nica cada registro em uma tabela, e chaves estrangeiras, que criam os vnculos entre as tabelas. Essa estrutura permite garantir a integridade dos dados, evitando inconsistncias e redundncias, e realizar consultas complexas de forma eficiente, utilizando a linguagem SQL (Structured Query Language)1F[2].

 

1.7.2                SGBDs No Relacionais: Flexibilidade e Escalabilidade

Os SGBDs no relacionais, tambm conhecidos como NoSQL (Not Only SQL), representam uma alternativa aos SGBDs relacionais, oferecendo modelos de dados mais flexveis e escalveis. Em vez de tabelas, eles utilizam diferentes estruturas para armazenar as informaes, como:

  • Chave-Valor: Armazenam dados em pares de chave e valor, de forma semelhante a um dicionrio, onde cada chave identifica um valor (ex: Redis).
  • Documento: Armazenam dados em documentos semiestruturados, como JSON ou XML, que podem conter diferentes tipos de informao e ter uma estrutura mais complexa (ex: MongoDB).
  • Coluna: Armazenam dados em colunas em vez de linhas, o que pode ser mais eficiente para consultas que envolvem a recuperao de colunas especficas (ex: Cassandra).
  • Grafo: Armazenam dados em ns e arestas, representando as relaes entre os dados de forma explcita, o que til para aplicaes que exploram conexes complexas (ex: Neo4j).

Essa flexibilidade permite que os SGBDs no relacionais se adaptem a diferentes tipos de dados, estruturados ou no estruturados, e que escalem horizontalmente para lidar com grandes volumes de informao e alta demanda de acesso. Eles podem ser especialmente teis em aplicaes de geoprocessamento que envolvem o processamento de grandes volumes de dados no estruturados, como dados de redes sociais, dados de sensores em tempo real ou dados de modelos de terreno complexos, onde a velocidade e a escalabilidade so mais importantes que a consistncia rgida dos dados.

No entanto, importante notar que os SGBDs no relacionais geralmente oferecem menos garantias de integridade e consistncia dos dados, possuem linguagens de consulta menos padronizadas e um conjunto de ferramentas e recursos menos maduro em comparao com os SGBDs relacionais.

1.8                    Banco de dados no contexto do geoprocessamento

Essa capacidade de estruturar e relacionar dados fundamental. Pense em um mapa digital: ele contm diversas camadas de informao, como estradas, rios, cidades, etc. Um SGBD relacional permite armazenar cada uma dessas camadas em uma tabela, definindo os atributos relevantes para cada elemento (nome da rua, tipo de rio, populao da cidade) e as relaes entre eles (uma cidade pertence a um estado, uma estrada cruza um rio).

No universo dos bancos de dados espaciais, existem diferentes caminhos que podem ser seguidos, e a escolha entre eles depende dos objetivos do projeto, da complexidade dos dados e das ferramentas disponveis.

Entre as opes mais utilizadas no geoprocessamento, destacam-se os bancos baseados em arquivos, como o GeoPackage (baseado em SQLite), e os sistemas mais robustos de banco de dados em rede, como o PostgreSQL com a extenso PostGIS. Ambos tm suas vantagens, e importante conhecer suas diferenas para fazer escolhas conscientes.

O GeoPackage uma alternativa leve, prtica e extremamente til para projetos de pequeno e mdio porte. 2                         MODELAGEM DE DADOS

A modelagem de dados um passo essencial na construo de qualquer banco de dados, pois garante a coerncia, a eficincia e a integridade das informaes armazenadas. No universo do geoprocessamento, em que frequentemente lidamos com conjuntos de dados complexos (vetoriais, raster e tabulares) e relaes espaciais (proximidade, adjacncia, sobreposio), uma boa modelagem permite que todas essas informaes sejam organizadas de forma consistente, evitando redundncias e erros. Neste captulo, vamos estudar mais a fundo o que modelagem, como definir entidades e atributos, de que modo surgem os relacionamentos, alm de explicar a questo da normalizao at a terceira forma normal, oferecendo tambm algumas reflexes sobre situaes em que pode ser til conhecer formas normais mais avanadas.

Quando falamos em modelagem de dados, temos em mente um processo de representar o mundo real ou o mundo do projeto dentro de um banco de dados. Em um projeto de geoprocessamento, por exemplo, precisamos refletir sobre quais so os objetos do mundo real (municpios, pontos de coleta, estradas, elementos de infraestrutura) e qual o contexto em que eles se relacionam. O termo entidades costuma representar esses objetos, que sero convertidos em tabelas. Cada tabela precisa ter colunas (atributos) que faro a descrio detalhada dos objetos, alm de definirmos as chaves que garantem a identificao nica e as referncias entre tabelas.

2.1                    Conceito Geral de Modelagem de Dados

A modelagem de dados corresponde ao desenho lgico do banco. como se estivssemos criando o mapa de onde ficaro armazenados determinados atributos, quais tabelas vo existir e como elas se relacionam. Fazemos isso para evitar falhas e retrabalhos, pois, sem um modelo definido, corre-se o risco de cada colaborador criar tabelas de maneira desconexa. Em geoprocessamento, isso pode resultar em camadas redundantes, tabelas com nomes confusos ou, pior, relaes espaciais que no fazem sentido (por exemplo, uma tabela de cidades que no faz referncia a estados ou regies).

Antes de escrever qualquer comando SQL de criao de tabelas, devemos pensar nas entidades principais. Se o projeto for sobre balneabilidade, por exemplo, faz sentido ter uma entidade que represente os pontos de coleta, outra que armazene as medies e, possivelmente, tabelas relacionadas a municpios e trechos de praia. A partir desse levantamento conceitual, comea-se a planejar os atributos relevantes: qual a chave primria da tabela de pontos de coleta, que tipo de dado utilizaremos para cada coluna (texto, nmero, data, geometria), e assim por diante.

2.2                   Entidades e Atributos no Contexto do Geoprocessamento

Quando pensamos em entidades, estamos falando de agrupamentos que se tornaro tabelas. O termo entidade vem da modelagem conceitual, algo que muitas vezes se descreve em um Diagrama Entidade-Relacionamento (DER). Cada entidade representa um tipo de objeto ou de fenmeno que iremos monitorar. Assim, uma entidade ponto_coleta pode ser traduzida em uma tabela cujo nome seja ponto_coleta ou algo similar. Cada registro dessa tabela ser um ponto especfico, com suas informaes. No QGIS ou no DB Browser, isso surgir como linhas de dados e colunas que descrevem cada ponto.

Atributos so as colunas de cada tabela. Ento, se a entidade ponto_coleta, faz sentido ter atributos como cdigo do ponto, nome ou descrio do local, coordenadas (latitude, longitude), municpio, data de criao do registro e assim por diante. importante pensar, j na modelagem, em qual formato de dado cada atributo ter. No geoprocessamento, podemos precisar armazenar valores geomtricos (um ponto, uma linha, um polgono) ou valores de referncia espacial. Em uma abordagem com SQLite e QGIS, podemos simplesmente guardar as geometrias em um formato suportado pelo QGIS (especialmente no GeoPackage). Se estivssemos no PostGIS, haveria a possibilidade de usar tipos especializados como geometry(Point, 4326).

2.3                   Relacionamentos: 1:1, 1:N e N:N

Um dos pontos mais delicados e importantes na modelagem diz respeito aos relacionamentos entre entidades. a que definimos como as informaes se ligam. Em termos gerais, existem trs tipos clssicos de relacionamentos:

 

-> 1:1 (um para um): Significa que cada registro de uma tabela se relaciona com, no mximo, um registro da outra tabela, e vice-versa. Esse tipo de relacionamento relativamente pouco comum em aplicaes de geoprocessamento, mas pode acontecer em cenrios muito especficos, como quando se divide uma tabela em duas partes por motivos de segurana ou desempenho. Em geral, se percebemos que duas entidades tm exatamente o mesmo nmero de registros, uma correspondncia exata, pode ser um sinal de que poderia ser uma nica tabela. Ainda assim, h casos em que manter duas tabelas separadas facilita a administrao ou a definio de permisses.

 

-> 1:N (um para muitos): Esse o mais frequente. Uma entidade ponto_coleta pode ter muitas medies. Isto , cada ponto de coleta existe apenas uma vez na tabela de pontos, mas pode aparecer muitas vezes na tabela que armazena os resultados de cada coleta. Em termos do DER, isso quer dizer que a tabela ponto_coleta possui uma chave primria (por exemplo, codigo_ponto), e a tabela medicao_balneabilidade referencia esse mesmo cdigo como chave estrangeira. Assim, todo registro de medio aponta para o seu ponto de coleta correspondente. Esse modelo se repete em inmeros contextos do geoprocessamento, como municpio (um) que contm bairros (muitos), zona de conservao (uma) que inclui parcelas de amostragem (muitas) e assim por diante.

 

-> N:N (muitos para muitos): Quando um registro em uma tabela pode se relacionar com diversos registros em outra, e vice-versa. Por exemplo, poderamos ter uma tabela pesquisador e outra projeto_de_pesquisa, onde cada pesquisador pode participar de vrios projetos e cada projeto pode ter vrios pesquisadores. Para representar isso, normalmente criamos uma tabela intermediria. Em problemas de geoprocessamento, um caso hipottico seria se cada rea de pesquisa (como uma UC Unidade de Conservao) pudesse ter diversos responsveis tcnicos, e cada responsvel tcnico pudesse estar vinculado a vrias reas de pesquisa. Nesse caso, faz-se uma terceira tabela que guarda as chaves primrias das outras duas, representando o relacionamento.

Desenhar e entender esses relacionamentos no incio evita confuses mais tarde. Em bancos relacionais, essa ligao formalizada por meio de chaves primrias e chaves estrangeiras, o que leva prxima questo: a integridade referencial.

2.4                   Chaves Primrias, Chaves Estrangeiras e Integridade Referencial

As chaves em um banco de dados relacional so elementos fundamentais para garantir a organizao e a integridade das informaes. Elas servem como mecanismos que evitam repeties indevidas, permitem a identificao nica de registros e estabelecem relaes entre diferentes tabelas.

As chaves primrias (PK, do ingls *Primary Key*) so campos que identificam de forma nica cada linha de uma tabela.

Por exemplo, se temos uma tabela chamada "ponto_coleta", o campo "codigo_ponto" pode ser a chave primria, desde que cada valor seja exclusivo. Isso significa que no ser possvel cadastrar dois pontos com o mesmo cdigo, o que garante a rastreabilidade e evita ambiguidades durante consultas, atualizaes ou anlises.

J as chaves estrangeiras (FK, de *Foreign Key*) so campos que criam ligaes entre tabelas diferentes. Uma chave estrangeira aponta para a chave primria de outra tabela, criando uma relao entre os dados.

Por exemplo, a tabela "medicao_balneabilidade" pode conter um campo tambm chamado "codigo_ponto", que serve para indicar a qual ponto de coleta aquela medio est associada. Nesse caso, "codigo_ponto" na tabela de medies uma chave estrangeira, pois se refere ao "codigo_ponto" da tabela de pontos. Essa ligao garante que no sejam inseridos registros de medies com cdigos de pontos inexistentes.

No caso do GeoPackage o formato de banco que usaremos no QGIS , ele utiliza internamente o sistema SQLite, que possui suporte a chaves estrangeiras. No entanto, o QGIS ainda apresenta limitaes na criao e gerenciamento dessas regras diretamente por sua interface grfica. Embora seja possvel ativar a verificao de integridade referencial no SQLite por meio do comando `PRAGMA foreign_keys = ON`, o QGIS nem sempre interpreta ou aplica essas regras de forma automtica.

Portanto, neste momento do curso, no criaremos as chaves primrias e estrangeiras usando comandos SQL diretamente. Em vez disso, focaremos nas ferramentas grficas disponveis no QGIS, compreendendo suas capacidades e restries. 2.5                   Normalizao de Dados

A normalizao um conjunto de tcnicas para organizar as colunas e as relaes das tabelas de forma a reduzir a redundncia e melhorar a integridade. Na prtica, a normalizao evita problemas do tipo informaes repetidas muitas vezes na mesma tabela, que acabam se desencontrando ou levando a dados inconsistentes.

De modo geral, discutimos formas normais (FN) que correspondem a graus de maturidade desse processo. Para a maior parte das aplicaes de geoprocessamento de nvel tcnico, chegar at a Terceira Forma Normal (3FN) costuma ser suficiente, mas vamos explicar cada uma:

 

 

 

2.5.1                Primeira Forma Normal (1FN):

A 1FN exige que cada atributo seja atmico, ou seja, nenhuma coluna armazene vrios valores ao mesmo tempo. Por exemplo, no se deve ter uma coluna lista_municipios em que se guardam vrios nomes de municpios separados por vrgula; o correto seria separar cada municpio em seu prprio registro ou, se existe relao, colocar em outra tabela ligada por chaves estrangeiras. Tambm se espera que cada tabela tenha uma chave primria. Em termos prticos, no geoprocessamento, isso significa evitar tabelas em que cada linha represente vrios polgonos, ou que tenhamos campos cumulativos do tipo coordenadas: (lat1, long1), (lat2, long2) quando na verdade cada feio deveria ser um registro independente ou uma geometria prpria.

2.5.2               Segunda Forma Normal (2FN):

Para estarmos em 2FN, a tabela antes precisa estar em 1FN e, adicionalmente, exige-se que todos os atributos no-chave dependam da chave primria por inteiro, especialmente quando a chave primria for composta. Isso evita a chamada dependncia parcial. Se, por exemplo, a chave primria for (codigo_ponto, data_medicao), todos os campos descritivos devem depender de ambos os valores, no apenas de um deles. Em bancos geoespaciais, essa regra ajuda a manter dados organizados quando h chaves compostas, como no caso de associar municpio e cdigo de coleta, ou associar data e trecho de rio.

2.5.3               Terceira Forma Normal (3FN):

Supondo que a tabela j esteja em 2FN, a 3FN requer que no existam dependncias transitivas, isto , um atributo no pode depender de outro atributo que no seja a chave primria. Em outras palavras, cada coluna no-chave deve depender diretamente da PK, e no de outra coluna intermediria. Isso evita situaes como ter um campo nome_municipio que dependa de uma outra coluna codigo_municipio que, por sua vez, depende da PK nesses casos, o melhor extrair esse nome_municipio para outra tabela (uma tabela de municpios) e fazer o relacionamento via chave estrangeira. No geoprocessamento, isso muito comum: ao invs de replicar vrias vezes o nome e o cdigo do municpio em diferentes tabelas, cria-se uma nica tabela municipio com o par (codigo, nome). Qualquer outra tabela que precise saber o nome do municpio faz referncia ao seu cdigo.

No contexto prtico de projetos de SIG, essas trs primeiras formas normais j ajudam muito, especialmente para dados alfanumricos. Entretanto, pode haver casos em que alguma forma de desnormalizao seja til para performance em consultas espaciais, desde que planejada. Portanto, o equilbrio entre uma boa normalizao e a eficincia do banco costuma ser parte do planejamento de projetos maiores.

2.6                   Consideraes sobre Formas Normais Avanadas

Para a maioria dos projetos tcnicos, atingir a 3FN suficiente. Entretanto, existem formas normais adicionais, como a BCNF (Forma Normal de Boyce-Codd) e a 4FN (Quarta Forma Normal), que tratam de restries mais especficas de dependncia. Geralmente, aplicam-se em cenrios complexos de bases de produo em larga escala, onde preciso ter um controle rigoroso de toda e qualquer redundncia. Em geoprocessamento, pode ocorrer a necessidade de atingir esse nvel de detalhe em projetos de grande porte, mas, em contextos educacionais e tcnicos, a 3FN costuma equilibrar bem clareza, consistncia e facilidade de implementao.

Em contrapartida, h projetos que deliberadamente fazem a desnormalizao de algumas tabelas, criando colunas redundantes para agilizar consultas espacialmente pesadas ou para evitar JOINs complexos em ndices de geometria. Em geral, isso requer um planejamento cuidadoso, porque facilita algumas operaes, mas pode dificultar a manuteno de dados. A deciso depende do tamanho do projeto, do nmero de usurios simultneos, das anlises espaciais realizadas e das exigncias de desempenho.

4.3                   Restries de Integridade: Garantindo a Qualidade dos Dados

Agora que sabemos como definir o tipo de informao que cada coluna ir guardar, precisamos estabelecer as regras do jogo. Em um banco de dados, essas regras so chamadas de restries de integridade (constraints). Elas funcionam como "guardas" que impedem a insero de dados invlidos ou inconsistentes, assegurando que a base de dados se mantenha confivel, organizada e lgica ao longo do tempo.

A aplicao de restries uma das prticas mais importantes na modelagem de um banco de dados profissional. Elas garantem que, independentemente de quem ou de qual software esteja inserindo os dados, as regras fundamentais do projeto sero sempre respeitadas. Vamos conhecer as principais restries utilizadas em SQL.

Principais Restries de Integridade

        NOT NULL (No Nulo)

o   O que faz: Garante que uma coluna no pode conter valores nulos (vazios). O preenchimento daquele campo se torna obrigatrio para cada novo registro inserido na tabela.

o   Quando usar: Em colunas cuja informao essencial para o registro fazer sentido.

o   Exemplo prtico: Na nossa tabela relatorio_balneabilidade, as colunas data_coleta e resultado_ecoli deveriam ser NOT NULL, pois um relatrio de balneabilidade sem a data da coleta ou sem o resultado da anlise no possui valor.

        UNIQUE (nico)

o   O que faz: Assegura que todos os valores em uma determinada coluna sejam diferentes uns dos outros. Nenhuma duplicao permitida.

o   Quando usar: Para campos que funcionam como um identificador que no pode se repetir, como o CPF de uma pessoa, o email de um usurio ou um cdigo de protocolo para uma amostra.

        PRIMARY KEY (Chave Primria)

o   O que faz: a restrio mais importante. Ela designa uma coluna (ou um conjunto de colunas) como o identificador nico e principal de cada linha da tabela. Uma chave primria , por definio, UNIQUE e NOT NULL.

o   Quando usar: Toda tabela bem modelada deve ter uma chave primria. Ela o "RG" de cada registro, permitindo que o banco localize, atualize ou remova uma linha especfica com mxima eficincia. tambm a base para criar relacionamentos com outras tabelas.

o   Exemplo prtico: O ideal para a nossa tabela relatorio_balneabilidade seria adicionar uma coluna, como id_relatorio do tipo INTEGER, e defini-la como PRIMARY KEY.

        FOREIGN KEY (Chave Estrangeira)

o   O que faz: Cria um vnculo lgico entre duas tabelas, garantindo a integridade referencial. Uma chave estrangeira em uma tabela "aponta" para a chave primria de outra tabela.

o   Quando usar: o pilar dos bancos de dados relacionais. Ela garante, por exemplo, que no seja possvel registrar um relatrio de medio para um ponto de coleta que no existe na tabela de locais.

o   Exemplo prtico: Se a nossa tabela locais_coleta tivesse uma chave primria id_ponto, poderamos adicionar uma coluna id_ponto_fk na tabela relatorio_balneabilidade e defini-la como uma FOREIGN KEY que referencia a tabela locais_coleta.

        CHECK (Verificar)

o   O que faz: Permite definir uma regra ou condio personalizada que o valor de uma coluna deve satisfazer para ser aceito.

o   Quando usar: Para aplicar regras de negcio especficas que no so cobertas por outras restries.

o   Exemplo prtico: Poderamos usar a restrio CHECK na coluna resultado_ecoli para garantir que o valor inserido seja sempre positivo ou zero (resultado_ecoli >= 0). Outro exemplo seria na coluna mare, para permitir apenas os valores 'Alta', 'Baixa' ou 'Reponto'.

A compreenso destas restries fundamental. Embora no QGIS possamos configurar formulrios e domnios para guiar o preenchimento, as restries SQL so mais robustas, pois so aplicadas diretamente no banco de dados, valendo para qualquer forma de acesso.

5.5.2.2            Combinando Condies com Operadores Lgicos

Muitas vezes, uma nica condio no suficiente para filtrar os dados da maneira que precisamos. Nesses casos, podemos combinar mltiplas condies usando operadores lgicos. Os principais operadores lgicos so AND, OR e NOT.

O operador AND usado para combinar duas ou mais condies. Para que um registro seja includo no resultado, todas as condies ligadas por AND devem ser verdadeiras.

Neste caso, um municpio s aparecer no resultado se AMBAS as condies (ser de SC E ter populao > 200.000) forem satisfeitas.

Operador OR

O operador OR tambm combina duas ou mais condies. No entanto, para um registro ser includo no resultado, pelo menos uma das condies ligadas por OR deve ser verdadeira.

        Sintaxe: WHERE condicao1 OR condicao2

Quando voc combina mltiplos operadores lgicos (AND, OR) em uma nica clusula WHERE, a ordem em que eles so avaliados pode afetar o resultado. FROM municipios

6.2.1                Restries via Formulrios de Atributos no QGIS

Contudo, fundamental compreender que essas restries pertencem ao projeto do QGIS e no estrutura do banco de dados em si. Em outras palavras, so restries no permanentes: elas funcionam apenas dentro do projeto onde foram configuradas. Caso esse banco de dados seja aberto em outro projeto do QGIS ou em um software diferente, as restries no estaro mais visveis ou ativas.

Portanto, ainda que o QGIS permita configurar formulrios que limitem os valores aceitos por campo (como no caso de listas suspensas), importante lembrar que tais validaes no fazem parte do banco. Isso significa que os dados podem ser editados fora do QGIS sem qualquer validao, comprometendo a integridade da informao se no houver regras aplicadas diretamente no banco de dados por meio de comandos SQL.

6.2.2               Validao de Dados no QGIS com Domnios de Campo

O QGIS possui um recurso chamado "Field Domains" (Domnios de Campo), que permite adicionar regras de validao para os dados diretamente dentro do banco de dados, especialmente em arquivos do tipo GeoPackage. Esse recurso utilizado para restringir os valores permitidos em determinados campos de uma tabela, de forma que os formulrios de atributos no QGIS possam guiar o usurio durante a edio dos dados.

Diferentemente das restries SQL (como CHECK ou FOREIGN KEY), os domnios de campo no so regras aplicadas por todo e qualquer software.

Eles funcionam dentro do ecossistema do QGIS, que os interpreta e os utiliza para configurar automaticamente os formulrios de atributos. Com isso, ao abrir uma tabela com domnio definido, o QGIS exibir listas suspensas e validadores que ajudam a prevenir erros de entrada.

Os domnios de campo podem ser de trs tipos principais:

                                    I.          Intervalo Numrico (Intervalo de domnio): Define um valor mnimo e mximo aceito para campos numricos (INTEGER, REAL). Exemplo: valores de temperatura entre 0 e 100 graus.

                                   II.          Valor Predefinido (domnio de valores codificados): Aplica-se a campos de texto. Define uma lista fechada de opes, como por exemplo 'Ausente', 'Fraca', 'Moderada', 'Forte'. Ao editar esse campo, o usurio ver uma lista suspensa com essas opes no formulrio de atributos.

                                 III.          Domnio Global (Globally Scoped)3F[5]: Pode ser aplicado a vrias tabelas dentro do mesmo banco de dados. Apesar do potencial de reutilizao, esse tipo de domnio limitado: no permite expressar relacionamentos entre tabelas, como garantir que um campo de uma tabela faa referncia a valores existentes em outra (como uma chave estrangeira faria).

possvel realizar algumas configuraes avanadas nos campos de domnios pois a janela de configurao de domnios de campo no QGIS oferece algumas opes:

Gerais:

a.     Valor Padro: Mantm o comportamento padro.

b.    Soma: Soma os valores.

c.     Mdia ponderada das Geometrias: Calcula a mdia ponderada dos valores em relao area.

Valores:

a.     Mnimo: Define o valor mnimo a ser aceito

b.    Mximo: define o valor mximo a ser aceito

Contudo, essencial entender que essa validao ocorre apenas no ambiente do QGIS. Qualquer manipulao dos dados feita fora dele no estar sujeita a essas regras. Para projetos mais exigentes ou multiusurios, onde a integridade dos dados crtica, altamente recomendada a utilizao de restries no prprio banco de dados, via comandos SQL.

De qualquer forma, os domnios de campo so uma excelente porta de entrada para boas prticas de validao de dados e podem ser utilizados em conjunto com os formulrios personalizados para criar uma experincia de edio mais segura e didaticamente orientada.

6.2.3               Validao de Dados via SQL

A validao de dados via SQL consiste em configurar regras de integridade diretamente no banco de dados, utilizando comandos como `CHECK`, `UNIQUE`, `NOT NULL` e chaves estrangeiras (`FOREIGN KEY`).

7.2.2               O Modelo de Dados Matricial (Raster)

Enquanto o modelo vetorial descreve o mundo atravs de objetos com fronteiras discretas, muitos fenmenos geogrficos, como a elevao de um terreno, a temperatura de uma rea ou uma imagem de satlite, so de natureza contnua.        O que faz: Utiliza as estatsticas internas do PostgreSQL (que so estimativas) para obter informaes sobre as tabelas, como o nmero de feies, em vez de fazer uma contagem exata.

       Quando usar: Ative esta opo para acelerar significativamente o carregamento de camadas de tabelas muito grandes (com milhes de registros). A contagem de feies pode no ser exata, mas o ganho de performance considervel.

     V.          Permitir salvar/carregar projetos QGIS no banco de dados

       O que faz: Habilita a funcionalidade de salvar o arquivo de projeto do QGIS (.qgz ou .qgs) diretamente dentro do banco de dados, em vez de em um arquivo local na sua mquina.

       Quando usar: Essencial para ambientes de trabalho colaborativos. Salvar o projeto no banco centraliza o acesso e garante que todos os membros da equipe abram a mesma verso do projeto, com as mesmas camadas e estilos.

   VI.          Permitir salvar/carregar metadados da camada QGIS no banco de dados

       O que faz: Permite que os estilos de visualizao das camadas, configuraes de formulrios e outros metadados especficos do QGIS sejam salvos diretamente no banco de dados.

       Quando usar: Quando esta opo usada em conjunto com a anterior, ela garante que, ao carregar o projeto do banco, todas as camadas aparecero com a simbologia, rtulos e estilos corretos para todos os usurios, promovendo uma padronizao visual completa.

  VII.          Mostre apenas camadas nos registrados de camada

       O que faz: Exibe somente as camadas que esto registradas na tabela de metadados do prprio PostGIS (geometry_columns e geography_columns).

       Quando usar: Esta opo geralmente deve ser mantida ativada, pois garante que o QGIS liste apenas as tabelas que so reconhecidas pelo PostGIS como camadas espaciais vlidas, proporcionando uma filtragem limpa e correta.

 

7.4.3               Conexo a um Banco de Dados em Rede (Servidor Remoto)

Em ambientes de trabalho colaborativos, comum que o banco de dados esteja em um servidor central na rede. O processo de conexo no QGIS quase idntico, alterando-se apenas os parmetros de acesso.

o   Host: Em vez de localhost, ser informado o endereo IP (ex: 192.168.0.100) ou o nome de domnio do servidor (ex: db.empresa.com.br). Esta informao fornecida pelo administrador do banco de dados ou da rede.

o   Porta: Geralmente 5432, a menos que o administrador a tenha alterado por motivos de segurana.

o   Credenciais: Sero fornecidas as credenciais de um usurio autorizado a acessar o banco de dados remotamente.

Observaes Importantes para Conexes em Rede:

o   Firewall: O servidor que hospeda o PostgreSQL deve ter regras de firewall que permitam o trfego de entrada na porta 5432 a partir da mquina do usurio.

o   Configurao do PostgreSQL (pg_hba.conf): O administrador do servidor precisa configurar o arquivo pg_hba.conf para autorizar explicitamente que o usurio se conecte a partir de um determinado endereo IP ou faixa de rede.

o   Segurana (SSL): Para conexes que atravessam redes no seguras (como a internet), fundamental utilizar um modo de conexo SSL para criptografar os dados entre o QGIS e o servidor.
